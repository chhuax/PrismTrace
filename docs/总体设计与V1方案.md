# AI 应用可观测性工具 - V1 设计

日期：2026-04-17
更新：2026-04-26
状态：进行中，已从 attach-first 调整为 observer-first，当前以 Codex 主线收口

## 1. 产品定义

本项目被定义为一个 AI 应用可观测性工具。

它不只是一个 prompt 嗅探器，不只是一个附加式调试器，也不是某一个 AI CLI 的单用途逆向工程工具。长期方向是围绕 AI 应用的观察、分析与解释构建产品。

第一步会更聚焦：

- 采集我们实际需要的高价值事实
- 在不重启被观测应用的前提下完成采集
- 保留当前会话，而不是强制开启新会话

第一阶段的简短定义是：

`在不中断当前会话的情况下，从正在运行的 AI 应用中捕获真正面向模型的信息，以便后续在这一事实层之上构建分析能力。`

截至 2026-04-26，这个定义需要再补一条当前已经落地的产品判断：

`PrismTrace 不再把 attach 当成产品主入口，而是转向以官方 observer 为主的事实采集路线。`

## 2. 产品路线图分层

产品应分三层演进：

### 第一层：信息采集

回答：我们能看到什么？

目标：

- 捕获真实请求载荷
- 在可见时捕获工具或 skill 注册信息
- 捕获响应流数据和有用的元数据
- 避免要求 SDK 集成或应用重启

### 第二层：会话重建

回答：发生了什么？

目标：

- 重建可用的会话时间线
- 将请求、响应与本地可观察事件关联起来
- 支持会话回放、对比和检查

### 第三层：分析与解释

回答：为什么会发生？

目标：

- 在可能时解释为什么 skill 或工具没有触发
- 解释 prompt 膨胀与上下文组成
- 识别回退、过滤、重复重试或路由切换等模式

## 3. 第一阶段范围

第一阶段只聚焦于为未来分析采集正确的事实。

我们不追求最大化捕获广度，而是优化那些对后续分析最有价值的信息。

### 需要捕获的核心数据

#### 请求元数据

- 时间戳
- 提供方
- 模型
- 端点
- 流式或非流式模式
- 延迟指标
- 在可见时记录 token 或大小估算

#### 请求载荷

- system prompt
- messages
- tools 或 functions 定义
- tool choice
- response format
- 采样参数
- 可见的附件描述或转换后的附件内容元数据

#### 响应数据

- 首 token 时间
- 完成时间
- 流 chunk 摘要
- 最终 assistant message
- tool calls
- finish reason
- 在可见时记录 usage 数据

#### 会话关联

- 进程 id
- 应用名称
- 在可见时记录命令或窗口上下文
- session id 或请求分组 id

#### 本地工具可见性

如果能从本地进程中看到，则记录：

- 候选工具或 skill 列表
- 最终发送给模型调用的已注册工具或 skill 列表

### 可选增强项

- prompt 组装前与组装后的快照
- 本地编排过滤日志
- 文件读写
- 命令执行事件
- 工作目录变化
- 用户交互事件
- 重试、回退与 provider 切换

### V1 的明确非目标

- 完整解释所有本地决策路径
- 保证支持所有基于 Node 的应用
- 完整覆盖多语言运行时
- 完美重建每一种多轮会话树

## 4. V1 用户体验

V1 应该采用 Rust 后端加本地 Web UI 的形态。

产品应更像一个本地可观测性控制台，而不是一个沉重的桌面壳应用。

### 用户流程

1. 在本地启动可观测性工具。
2. 看到统一的 source / runtime 列表。
3. 对已接通的 observer source 点击 Observe。
4. 在不中断当前运行会话的情况下开始捕获新的高层事件，并在可见时补充请求事实。
5. 实时查看会话时间线。
6. 打开某个请求或事件，检查其 payload、响应、工具和元数据。

### 请求详情视图

每个被捕获的请求应展示：

- 原始请求载荷
- 响应摘要与流信息
- 可见的工具或 skills
- 元数据以及它在时间线中的位置

### 为什么选择本地 Web UI

- 相比 CLI，更适合检查大体积 payload
- 比沉重的 Electron 壳更容易快速交付
- Rust 系统后端与 UI 渲染之间边界清晰
- 更容易扩展到过滤、diff 和未来分析视图

## 5. 技术架构

系统应被拆分为五层，并保持清晰边界。

### 5.1 Source 接入层

这一层在代码层目前仍包含两类入口：

- attach source
- observer source

其中 observer 是当前产品主线；attach 已不再作为默认产品方案或默认 CLI 入口，`prismtrace-host` 中对外暴露的 attach 控制链也已清理。

### 5.1.1 Attach 子层

职责：

- 发现候选进程
- 附加到目标进程
- 注入或激活探针
- 管理连接生命周期
- 从 detach 或失败中恢复

### 5.1.2 Observer 子层

职责：

- 发现 observer source
- 建立握手
- 读取高层事件
- 归一化为统一 observability event
- 将结果落到统一 artifact / timeline 读链

### 5.2 Probe 层

这一层运行在最靠近目标进程的位置。

职责：

- hook 关键的 Node 或 Electron 网络路径
- 在加密和传输之前捕获请求载荷
- 捕获响应以及流 chunk
- 在可能时捕获可见的工具或 skill 信息

这一层只负责收集事实，不负责复杂分析。

### 5.3 事件管线

职责：

- 规范化原始探针事件
- 配对请求与响应
- 将事件关联到会话中
- 按时间线排序事件
- 应用截断或脱敏策略

输出：

- 一个稳定的可观测性事件 schema

### 5.4 存储层

V1 的存储可以保持简单且本地化。

推荐起点：

- 使用 SQLite 存储元数据、索引和会话关系
- 使用 SQLite 或 blob 文件存储大型 payload 正文

必需能力：

- 按应用、时间、provider、model 或 session 过滤
- 搜索 payload 内容
- 为未来分析模块提供稳定的数据访问

### 5.5 分析与 UI 层

V1 应先从浏览和搜索能力开始，但架构必须为未来分析预留空间。

计划中的未来能力：

- prompt diff
- 工具可见性 diff
- 失败模式归因
- skill 未触发诊断
- token、延迟和成本分析

### 架构规则

`Probe 负责收集事实，Analysis 负责解释事实。`

这一边界很重要，因为它能让第一阶段专注于可靠采集，而不会过早承诺解释能力。

## 6. 当前产品边界

当前达成一致的边界：

- 第一版仅支持 macOS
- 聚焦基于 Node 或 Electron 的 AI CLI 与桌面应用
- 不要求重启被观测应用
- 以 payload 可见性为首要价值
- 为后续分析与解释功能留出空间

同时新增一条当前已明确的产品边界：

- 采集面按 `source` 而不是按 `attach` 统一建模
- 当前产品主线以 observer 为主，优先接入存在官方或准官方运行时观测入口的目标
- attach 不再作为当前产品方案和默认 CLI 入口，`prismtrace-host` 中旧 attach/readiness 控制实现已清理

## 7. 开放设计议题

下一个需要设计的主要议题是：

- V1 首先应针对哪些具体的 Node 或 Electron hook 点

这个决策将决定第一版实现是否可行，以及初始兼容性能够覆盖多广。

## 8. 设计五：首批 Node 与 Electron Hook 目标

V1 应优先选择那些能在基于 Node 的 AI 应用中最大化 payload 可见性，同时又能保持实现现实可行的 hook 点。

目标不是 hook 每一层抽象，而是在尽量少的高杠杆位置上，尽可能可靠地捕获面向模型的 payload。

### 8.1 Hook 策略原则

对 V1 来说，最好的默认规则是：

`在尽可能高的层级进行 hook，以保留语义；在必要时再下探到底层，以保留兼容性。`

这意味着：

- 优先选择仍然暴露结构化请求对象的拦截点
- 只有在高层 SDK hook 不存在时，才回退到更底层的 HTTP 层拦截
- 避免从 TLS 或 socket 层起步，因为那样更难重建 payload 语义

### 8.2 V1 的优先顺序

第一版应按以下顺序选择 hook 点：

1. 全局 `fetch`
2. `undici`
3. Node `http` 与 `https`
4. 在低成本且高价值时，加入选择性的 SDK 层 hook
5. 后续可选支持 Electron renderer 与 main-process 桥接

这一顺序可以在不过早过度特化的前提下，为现代 Node 与 Electron AI 工具带来良好覆盖。

### 8.3 第一层级：全局 Fetch

许多现代 Node 应用与 Electron 应用都会通过 `fetch` 发送模型请求，要么直接使用，要么通过封装层使用。

为什么它重要：

- 覆盖范围广
- 请求体附近仍保留较强的语义结构
- 相比更深的传输层 hook，实现复杂度更低
- 一个拦截点就能捕获多个 provider

需要捕获的内容：

- 请求 URL
- 方法
- 本地组装后、发送前的 headers
- 加密前的请求体
- 响应状态码和 headers
- 在存在时捕获流式响应体 chunk

主要限制：

- 如果应用完全绕过 `fetch`，这个 hook 就看不到任何东西

### 8.4 第二层级：Undici

`undici` 是一个特别重要的目标，因为许多 Node 运行时与现代库都会直接或间接使用它。

为什么它重要：

- 在当前 Node 生态中覆盖强
- 比通用封装更接近真实请求路径
- 当应用代码没有暴露简单的全局 `fetch` 路径时，它是有用的回退点

可能有价值的拦截点：

- `fetch`
- `request`
- `stream`
- dispatcher 层的请求执行

需要捕获的内容：

- 规范化后的请求载荷
- 流生命周期事件
- 请求开始、首字节、结束等时间标记

### 8.5 第三层级：Node HTTP 与 HTTPS

这一层是为那些没有以易于 hook 的方式使用 `fetch` 或 `undici` 的工具提供兼容性兜底。

为什么它重要：

- 兜底覆盖面广
- 可以捕获较旧 SDK 和手写客户端
- 能提高在异构工具中观察到 provider 请求的概率

需要捕获的内容：

- 主机名与路径
- headers
- flush 之前的 body writes
- 响应 chunks

主要权衡：

- 这里的语义弱于更高层的 hook
- 要重建干净的 JSON payload，可能需要缓冲和重组

### 8.6 第四层级：选择性 SDK Hook

V1 不应依赖 SDK 特定 hook，但在成本低时，做选择性支持可以增加价值。

示例：

- OpenAI 客户端封装
- Anthropic 客户端封装
- 目标应用内部的 provider 抽象层

为什么这有用：

- 能保留更多富含意图的信息
- 可能暴露工具注册或预处理转换过程
- 可能揭示最终序列化之前的请求构建阶段

为什么它不应成为默认基础：

- 跨版本脆弱
- 维护成本高
- 很容易过度拟合某一个工具

V1 推荐规则：

- 只有在传输层 hook 已经工作后，才把 SDK hook 作为定向增强

### 8.7 Electron 特有考虑

Electron 应用可能从以下位置发送模型请求：

- main process
- renderer process
- 启用了 Node 的 preload bridge

V1 应假设 payload 生成逻辑可能存在于这些任意位置。

推荐做法：

- 先聚焦 main process 以及任何具备 Node 能力的运行时路径
- 将纯 renderer 的浏览器路径视为次级目标
- 记录进程身份，以便之后把事件关联回正确的应用组件

这意味着 V1 应优先保证附加可靠性，而不是追求对 renderer 的完整内省。

### 8.8 捕获工具与 Skill 可见性

捕获 prompt payload 是首要目标，但当成本足够低时，V1 也应机会式地捕获本地工具或 skill 的可见性。

适合作为候选的 hook 类别：

- 将工具定义组装进模型请求的函数
- 生成可调用工具描述符的注册表
- 把内部工具翻译成 provider 特定 schema 的包装函数

如果可见，应记录：

- 候选工具数量与名称
- 被过滤后工具数量与名称
- 请求中最终包含的序列化工具 schema

重要边界：

- V1 不需要证明某个工具或 skill 为什么被排除
- V1 只需要保留足够证据，以便未来分析

### 8.9 V1 不应从这些层开始

V1 应明确避免从以下层开始：

- 原始 TLS 拦截
- 将抓包作为主要方法
- 在磁盘上重写 JavaScript 源码
- 把针对单应用的自定义逆向工程作为默认路径

这些方法未来可能在特定场景中有帮助，但它们并不是一个可维护可观测性产品的正确默认基础。

### 8.10 推荐的 V1 Hook 组合

推荐的第一版实现组合为：

- 主路径：全局 `fetch`
- 主路径：`undici`
- 兜底：`http` 与 `https`
- 增强项：请求体 JSON 重建与响应流捕获
- 机会式增强：在自然暴露时记录本地工具或 skill 的组装可见性

这一组合最符合当前产品边界：

- 仅 macOS
- Node 与 Electron 目标
- 无需重启
- 以 payload 可见性优先

## 9. 下一个设计议题

在 hook 目标之后，下一个设计议题应是：

- 对已在运行的 Node 与 Electron 进程，macOS 上的附加与注入该如何工作

## 10. 设计六：macOS 上的附加与注入

V1 需要一种现实可行的方式，能够在 macOS 上附加到一个已经运行中的 Node 或 Electron 进程，注入探针，并在不重启目标的情况下，把捕获到的事件流回本地可观测性控制台。

这是整个产品的技术铰链点。

### 10.1 设计目标

附加系统必须满足以下要求：

- 能附加到已经运行的目标进程
- 避免重启目标应用
- 将探针注入到正在运行的 runtime 中
- 在附加后捕获未来请求
- 将事件流回 Rust 后端
- 可干净地 detach，不导致崩溃，也不永久修改目标

重要边界：

- V1 只保证观测附加之后发生的活动
- V1 不承诺追溯恢复探针注入前已经发出的请求

### 10.2 候选方案

这里有三种宽泛的实现路径。

#### 方案 A：构建自定义原生注入器

这意味着使用更底层的 macOS 调试或任务控制原语，自行实现附加与注入机制。

优点：

- 控制力最大
- 不依赖第三方运行时插桩框架
- 长期看可能在性能和打包上有优势

缺点：

- 实现风险最高
- 跨 macOS 版本更脆弱
- 在拿到第一批可用 payload 捕获之前，需要投入更多工作
- 很可能过早把项目拖入操作系统内部细节

V1 建议：

- 不要从这里开始

#### 方案 B：使用现有的动态插桩运行时

这意味着依赖一个成熟的运行时插桩后端，并在其之上构建我们自己的探针与事件管线。

优点：

- 最快能得到可用原型
- 天然贴合实时附加工作流
- 降低一开始就编写不稳定注入代码的风险

缺点：

- 为架构引入一个外部依赖
- 可能会对打包或支持环境带来限制
- 可能需要为长期产品化加固做额外适配

V1 建议：

- 从这里开始

#### 方案 C：基于调试器的附加

这意味着使用调试器式工作流来暂停、检查，并可能向运行中的进程注入行为。

优点：

- 在 macOS 开发机上概念上是可行的
- 对诊断和实验有帮助

缺点：

- 不是持续运行时观测的良好产品基础
- 操作摩擦更高
- 更难打磨成顺滑、常驻的用户体验

V1 建议：

- 只把它作为开发辅助，而不是核心产品路径

### 10.3 V1 的推荐方向

V1 应使用现有的动态插桩运行时作为附加后端，同时把架构其余部分掌握在自己手中。

这意味着：

- Rust 负责进程发现、探针生命周期、事件规范化、存储和 UI
- 插桩后端负责底层实时附加和进程内执行
- 注入的探针逻辑保持产品专属，并聚焦 Node 与 Electron payload 捕获

这样能为项目提供一个现实可行的第一步：

- 我们能验证 prompt payload 捕获是否真的可行
- 我们能了解哪些 Node 与 Electron 目标具备兼容性
- 我们可以避免把第一个里程碑花在自定义 OS 注入机制上

### 10.4 附加流程

V1 推荐的附加流程如下：

1. 发现候选进程。
2. 识别可能的 Node 或 Electron 目标。
3. 通过插桩后端尝试实时附加。
4. 注入一个轻量级 bootstrap probe。
5. bootstrap 安装选定的 hook 组合。
6. probe 打开回连 Rust host 的消息通道。
7. 捕获到的事件流入事件管线。
8. 用户可以 detach，之后移除 hook 或禁用 probe。

这样可以让附加管线保持显式且易于调试。

### 10.5 Probe Bootstrap 职责

运行在目标进程内的 bootstrap 应尽可能小。

职责：

- 检测运行时形态
- 检测 `fetch`、`undici` 或 `http/https` 是否可用
- 安装正确的 hook 集合
- 避免重复安装
- 向 host 暴露健康状态和版本信息
- 通过稳定桥接通道发送事件流

bootstrap 不应做重量级分析或本地持久化。

### 10.6 Host 与 Probe 通信

host 与 probe 之间需要一个通信通道，用于传递：

- 健康状态
- 捕获到的事件
- hook 安装状态
- detach 或 shutdown 命令

V1 应保持简单：

- 面向行或带帧的 JSON 消息已经足够
- 事件顺序应在 probe 侧打时间戳，并由 host 进行规范化
- 桥接层应能容忍突发式流输出

host 应将 probe 视为不可信的事件生产者，并在写入存储前验证消息结构。

### 10.7 macOS 上的目标可附加性

在 macOS 上，并不是每个进程都能在实践中被附加。

V1 应假设：

- 用户态的 Node 与 Electron 应用是主要支持目标
- 受系统保护的进程可能不可用
- 有些应用可能因为运行时加固、签名或环境特定限制而拒绝附加
- 用户可能需要具备普通 macOS 开发者环境下的相关权限，才能完成实时插桩工作流

这意味着，附加支持应以兼容性矩阵的形式呈现，而不是笼统承诺。

### 10.8 失败模型

在一个有用的产品里，附加不是简单的二元成功或失败。V1 应清晰分类失败类型。

推荐的失败类别：

- 进程不受支持
- 权限被拒绝
- 识别到运行时，但 hook 点不可用
- 探针已注入，但事件通道失败
- hook 已安装，但 payload 重建不完整

这很重要，因为产品本身就是可观测性软件。它应解释自己能观察到什么、观察不到什么。

### 10.9 注入安全规则

探针必须保持保守行为。

规则：

- 永远不要修改磁盘上的应用文件
- 避免永久性运行时变更
- 让 hook 安装保持幂等
- 对捕获的 payload 大小进行边界控制
- 当运行时形态未知时，采用失败即关闭的策略
- 支持快速 detach 和禁用 probe

探针应优先接受“看不见”，而不是冒着破坏目标应用稳定性的风险。

### 10.10 打包影响

由于附加是最难的部分，V1 的打包结构应把产品价值与附加后端复杂度分离开来。

推荐结构：

- Rust 核心二进制
- 本地 Web UI
- 可插拔的插桩后端适配器
- 面向 Node 与 Electron 目标的版本化注入探针包

这样团队未来就能在不重写存储、事件模型或 UI 的情况下演进附加后端。

### 10.11 V1 建议总结

对于 V1：

- 使用动态插桩后端，而不是先自研原生注入器
- 只支持已在运行的 macOS 用户态 Node 与 Electron 目标
- 只捕获附加之后发生的活动
- 保持 bootstrap probe 轻量
- 让 host 主导事件整形与持久化
- 诚实且明确地呈现兼容性边界

## 11. 下一个设计议题

在附加与注入之后，下一个设计议题应是：

- 请求、响应、工具可见性和会话关联的具体事件 schema

## 12. 设计七：事件 Schema

事件 schema 是整个产品的基础。

如果这一层过窄，后续分析就不可能成立；如果这一层过于松散，系统就会变得难以查询、比较和推理。

V1 应定义一个稳定、面向事实的 schema，用来捕获“发生了什么”，而不强行引入过早解释。

### 12.1 Schema 设计原则

schema 应遵循以下规则：

- 先存事实，再下结论
- 同时保留传输细节和语义 payload 细节
- 将原始捕获数据与规范化字段分开
- 支持在 hook 只能看到部分信息时的部分捕获
- 保留足够的关联信息，以重建会话时间线

重要规则：

- schema 必须能容忍可见性不完整，而不会破坏整个事件模型

### 12.2 顶层对象模型

V1 应使用四个主要实体：

- `process_instance`
- `session`
- `event`
- `artifact`

对 V1 来说，这已经足够，无需过早过度建模。

### 12.3 Process Instance

`process_instance` 表示在某个时间段内被观察的一个正在运行的进程。

建议字段：

- `process_instance_id`
- `pid`
- `parent_pid`
- `app_name`
- `executable_path`
- `runtime_kind`
- `host_machine_id`
- `attach_started_at`
- `attach_ended_at`
- `attach_status`
- `probe_version`

目的：

- 标识是哪一个运行中目标产生了一组事件
- 区分同一个应用在不同时间上的多次附加

### 12.4 Session

`session` 表示一组逻辑上相关的 AI 活动。

一个 session 可能映射到：

- 一次 CLI 对话
- 一个桌面聊天标签页
- 从上下文中发现的一条请求链
- 或者在只有部分信号时，构造出的一个合成分组

建议字段：

- `session_id`
- `process_instance_id`
- `session_key`
- `session_source`
- `title`
- `created_at`
- `last_seen_at`
- `status`

说明：

- 如果可见，`session_key` 可以来自应用暴露的标识符
- `session_source` 应记录该分组是来自原生应用元数据、请求关联，还是启发式重建

### 12.5 Event

`event` 是系统中的主要事实记录。

每一个被捕获的观察结果都应以事件的形式进入系统。

建议的通用字段：

- `event_id`
- `session_id`
- `process_instance_id`
- `event_type`
- `captured_at`
- `observed_at`
- `sequence_no`
- `source_layer`
- `capture_confidence`
- `partial`
- `raw_ref`
- `normalized_ref`

字段意图：

- `captured_at` 表示 probe 发出事件的时间
- `observed_at` 表示底层运行时动作真正发生的时间（如果已知）
- `sequence_no` 用于帮助维持本地顺序
- `source_layer` 指出事件来自 `fetch`、`undici`、`http`、SDK hook 或其他来源
- `capture_confidence` 允许后续分析对证据质量进行排序
- `partial` 告诉系统不要假设它看到了完整全景

### 12.6 事件类型

V1 应支持以下初始事件类型：

- `request_started`
- `request_payload_captured`
- `response_headers_received`
- `response_stream_chunk`
- `response_completed`
- `tool_visibility_snapshot`
- `session_metadata_updated`
- `probe_status`
- `error_observed`

这个列表是有意保持精简的。它足以重建大多数有价值的请求生命周期，而不会让管线过度复杂化。

### 12.7 请求事件载荷

对于请求相关事件，规范化后的 payload 应支持以下字段：

- `request_id`
- `provider`
- `endpoint`
- `model`
- `method`
- `stream`
- `headers_visible`
- `payload_format`
- `system_text`
- `messages_json`
- `tools_json`
- `tool_choice`
- `response_format`
- `sampling_json`
- `attachment_summary_json`
- `body_size_bytes`

说明：

- `messages_json` 和 `tools_json` 应尽可能保留原始结构
- 后续可为分析增加规范化辅助字段，但对 V1 来说，原始语义更重要

### 12.8 响应事件载荷

对于响应相关事件，规范化后的 payload 应支持以下字段：

- `request_id`
- `status_code`
- `response_headers_visible`
- `first_byte_at`
- `completed_at`
- `finish_reason`
- `usage_json`
- `assistant_message_json`
- `tool_calls_json`
- `stream_summary_json`
- `body_size_bytes`

schema 应允许响应事件以增量方式到达，并在之后由事件管线合并。

### 12.9 工具可见性快照

该事件用于捕获在可观察时，本地可见的工具或 skills。

建议字段：

- `request_id`
- `visibility_stage`
- `candidate_tools_json`
- `filtered_tools_json`
- `final_tools_json`
- `tool_count_candidate`
- `tool_count_filtered`
- `tool_count_final`

`visibility_stage` 应描述该快照来自哪个阶段，例如：

- pre-filter
- post-filter
- pre-serialization
- request-embedded

这很重要，因为产品未来会分析候选工具与最终模型可见工具之间的差异。

### 12.10 Probe Status Event

probe 健康状态也应被建模为事件，而不是隐藏在日志中。

建议字段：

- `probe_state`
- `hook_bundle_version`
- `available_hooks_json`
- `installed_hooks_json`
- `failed_hooks_json`
- `error_message`

这样 UI 就能解释系统实际能观察到什么，以及遗漏了什么。

### 12.11 Error Observed Event

错误应该是一等事实，因为后续分析会需要它们。

建议字段：

- `error_class`
- `error_scope`
- `request_id`
- `message`
- `details_json`
- `recoverable`

可能的 scope：

- attach
- hook_installation
- request_capture
- response_capture
- session_correlation

### 12.12 Artifact

大型原始 payload 和流片段不应总是内联存放在每一行事件中。

`artifact` 用于承载更大的正文或原始捕获内容。

建议字段：

- `artifact_id`
- `artifact_type`
- `storage_kind`
- `storage_path`
- `content_type`
- `size_bytes`
- `sha256`
- `created_at`

示例：

- 原始请求体
- 原始响应 chunk 流
- 脱敏后的 payload 副本
- hook 诊断转储

### 12.13 原始存储与规范化存储

每一次重要捕获都应同时保留：

- 原始表示
- 规范化表示

二者都重要的原因：

- 原始数据对取证可信度和未来重处理是必要的
- 规范化数据对过滤、UI 和分析是必要的

V1 不应强制要求原始形式与规范化形式完全一致。

### 12.14 关联键

事件管线需要稳定的方法来连接相关事件。

在可见时，V1 应支持以下关联键：

- `request_id`
- `session_id`
- `process_instance_id`
- `provider request fingerprint`
- `hook-local sequence id`

host 应接受这样一个事实：有些事件在一开始只能以概率方式关联。

### 12.15 脱敏准备

即使 V1 还没有完整实现脱敏策略，schema 也应提前为其做好准备。

建议支持字段：

- `contains_sensitive_data`
- `redaction_status`
- `redaction_policy_version`

这样在以后加入隐私控制时，就不需要重新设计存储层。

### 12.16 Schema 的查询优先级

schema 应使以下 V1 查询足够容易：

- 展示某个进程的全部请求
- 按时间顺序展示某个 session 中的全部事件
- 打开某个请求，左右并排检查请求与响应
- 比较多个请求中最终嵌入的工具
- 搜索特定 prompt 片段或模型名
- 按 source layer 或 hook 类型检查失败

如果 schema 能很好地支持这些查询，那它就完成了自己的职责。

## 13. 下一个设计议题

在事件 schema 之后，下一个设计议题应是：

- 事件、artifact、索引与搜索的存储布局和本地数据库设计

## 14. 设计八：存储布局与本地数据库设计

V1 应采用 local-first 的存储设计，既足够简单以便快速交付，又足够强大以支持后续分析能力。

存储设计应优化以下方面：

- 以追加为主的事件写入
- 时间线重建
- 大型请求与响应正文的检查
- 过滤与搜索
- 在不重写原始事实层的前提下，后续加入分析结果

### 14.1 存储模型

V1 应采用混合式本地存储模型：

- 使用 SQLite 存储结构化元数据、索引、关系和可查询的规范化字段
- 在需要时，用文件承载大型原始 payload 和高 chunk 密度的正文

这样可以取得实用平衡：

- SQLite 仍然是查询和 UI 视图的事实来源
- 大型 payload 正文不会拖慢所有热点查询路径

### 14.2 推荐的本地数据布局

V1 应将所有本地数据放在产品的一个工作区根目录下，例如：

- `state/observability.db`
- `state/artifacts/<artifact_id>`
- `state/tmp/`
- `state/logs/`

具体路径可以稍后再定，但结构应把以下内容分离：

- 持久化、可查询的数据
- 大型不可变 artifacts
- 临时写入文件
- 运维日志

### 14.3 SQLite 作为元数据骨架

SQLite 应保存：

- process instances
- sessions
- events
- artifact metadata
- 搜索索引
- 未来的分析输出

为什么 SQLite 适合作为 V1：

- 易于本地部署
- 对关系型检索和过滤支持优秀
- 对早期版本性能足够
- 适合 macOS 本地工具的简单打包

V1 应避免仅为存储而引入单独的本地服务。

### 14.4 建议的核心表

第一版数据库形态应包括以下表：

- `process_instances`
- `sessions`
- `events`
- `artifacts`
- `event_artifacts`
- `session_labels`

如果有用，也可以较早加入的可选表：

- `probe_health_snapshots`
- `ingestion_failures`

### 14.5 Events 表策略

`events` 表应为时间线和过滤查询优化。

推荐列分组：

- 标识：`event_id`、`session_id`、`process_instance_id`
- 排序：`captured_at`、`observed_at`、`sequence_no`
- 分类：`event_type`、`source_layer`
- 关联：`request_id`、`provider`、`model`
- 状态：`partial`、`capture_confidence`
- 用于快速过滤的紧凑规范化字段
- 在合适时保存 `raw_ref` 与 `normalized_ref` 指针

重要设计规则：

- 如果 UI 需要频繁按字段过滤，就不要把整个规范化 payload 都塞进一个不透明 JSON 列

相反，应当：

- 将高价值过滤键保存在独立列中
- 将丰富的嵌套结构保存在 JSON 中

### 14.6 Artifacts 表策略

`artifacts` 表应索引那些单独存储的大型或原始捕获正文。

推荐用途：

- 原始请求载荷
- 原始流式响应
- 脱敏副本
- 调试捕获片段

推荐元数据：

- `artifact_id`
- `artifact_type`
- `storage_kind`
- `storage_path`
- `content_type`
- `size_bytes`
- `sha256`
- `compression`
- `created_at`

V1 一旦写入 artifact 文件，就应倾向于保持其不可变。

### 14.7 何时内联，何时外置 payload

V1 应采用简单策略：

- 小型规范化 payload 可以留在 SQLite 中
- 大型原始正文应移入 artifact 存储
- 流量大的流式捕获几乎总应写入 artifacts

这样既能保持交互式查询快速，又能保留完整证据。

### 14.8 搜索策略

V1 应从一开始就支持有价值的本地搜索。

推荐的搜索层：

- 在 SQLite 中用精确过滤列支持 provider、model、app、source layer、status 与时间
- 对 prompt 片段、消息内容和工具名提供全文搜索

V1 的务实做法：

- 对选定文本字段使用 SQLite FTS
- 索引规范化后的 prompt 可见内容，而不是每一个原始字节

这样无需引入独立搜索引擎，也能带来真实产品价值。

### 14.9 写入路径

写入路径应以追加为导向，并具备韧性。

推荐写入流程：

1. host 接收 probe 事件
2. 校验事件
3. 在需要时将大正文写入 artifact 存储
4. 插入规范化事件行
5. 更新搜索索引
6. 发出 UI 通知

关键属性：

- 写入流程不应被重量级分析阻塞

这意味着 V1 应将以下两者分离：

- 事件持久化
- 派生分析任务

### 14.10 读取路径

读取路径应高效支持以下产品视图：

- 已附加目标的活动流
- 按 session 展示的时间线
- 按请求查看的详情面板
- 过滤后的请求列表
- prompt 搜索结果

对常见列表视图来说，UI 不应每次都从头重建完整响应正文。

这意味着存储层应维护：

- 轻量级摘要字段
- 指向完整 payload 正文的直接指针

### 14.11 摘要字段

V1 应在写入时计算并存储摘要字段，以支持快速浏览。

有用示例：

- 请求预览文本
- 响应预览文本
- 工具数量
- stream 标记
- 正文大小
- finish reason
- 请求耗时
- 应用名称

这些字段不是原始数据的替代品，它们的存在是为了让 UI 保持响应迅速。

### 14.12 索引优先级

第一批索引应围绕真实产品使用场景优化。

推荐索引：

- 按 `session_id` 与时间索引 events
- 按 `process_instance_id` 与时间索引 events
- 按 `request_id` 索引 events
- 按 `provider`、`model` 与时间索引 events
- 按 `last_seen_at` 索引 sessions
- 按 `artifact_id` 索引 artifacts

FTS 应覆盖：

- 请求 prompt 文本
- 响应文本
- 工具名称
- 在有用时覆盖模型名和 provider 标签

### 14.13 保留与清理

即使在 V1，存储增长也需要策略。

推荐默认值：

- 元数据保留时间长于大型原始 artifacts
- 支持按 session 或时间范围手动清理
- 为未来基于脱敏的分层保留策略做好准备

V1 不需要自动激进清理，但也不能假设本地磁盘无限大。

### 14.14 迁移策略

schema 会演进，因此 V1 应从一开始就考虑迁移。

要求：

- 明确的数据库 schema 版本
- 仅向前迁移
- 在事件旁记录 probe 版本
- 启动时做存储兼容性检查

这很重要，因为随着兼容性提升，事件模型与规范化逻辑都会变化。

### 14.15 隐私与本地性

由于本产品会捕获敏感 AI payload，存储设计应默认把隐私视为重要前提。

V1 的存储立场：

- 默认本地
- 不依赖云
- 没有隐藏的后台导出
- 明确区分原始捕获内容与脱敏后或派生内容

如果实现清晰，这会成为产品的信任优势。

### 14.16 推荐的 V1 存储总结

V1 应使用：

- SQLite 存储结构化事件与会话数据
- SQLite FTS 进行 prompt 与响应搜索
- 外部 artifact 文件存储大型原始 payload
- 采用追加式写入并预计算摘要
- 提供明确的 schema 版本控制与迁移支持

这已经足以支撑可观测性控制台、未来分析模块以及 local-first 的信任模型。

## 15. 下一个设计议题

在存储之后，下一个设计议题应是：

- 脱敏、隐私控制以及对敏感捕获 payload 的安全处理

## 16. 设计十：最小可行实现计划

V1 的目的不是证明产品在理论上的一切能力。

V1 的目的是从零走到一个真实、可用的本地可观测性工作流，让用户可以：

- 看到一个正在运行的 Node 或 Electron AI 目标
- 在不重启的情况下附加到它
- 捕获新的面向模型的 payload
- 在本地控制台中检查这些 payload

因此，这份实现计划应优化“用最短路径拿到可信信号”。

### 16.1 V1 成功标准

如果用户在 macOS 上可以做到以下事情，V1 就是成功的：

1. 启动工具
2. 识别出一个受支持的运行中 AI 进程
3. 成功附加到它
4. 捕获至少一个真实的、附加后产生的 LLM 请求
5. 在 UI 中检查请求载荷、响应摘要和元数据

其他能力都很有价值，但属于次要项。

### 16.2 里程碑结构

推荐的实现应拆分为六个里程碑：

1. 骨架平台
2. attach 与 observer 双入口
3. payload / event 捕获管线
4. 本地可观测性控制台
5. 会话关联与搜索
6. 加固与兼容性打磨

这样能让团队专注于交付可工作的切片，而不是在前期堆太多基础设施。

### 16.3 里程碑 1：骨架平台

状态：

- 已完成

目标：

- 在尚未解决深层插桩问题前，先建立 host 应用的整体形态

交付物：

- 一个 Rust workspace，清晰拆分 host、storage、event model 和 UI server 的 crate 或模块
- 本地状态路径的配置模型
- SQLite schema bootstrap 与 migration runner
- macOS 上的进程发现能力
- 占位用的 attach 状态与健康上报
- 一个本地 Web UI 壳，提供空的 process list、sessions 和 requests 视图

为什么这个里程碑重要：

- 它定义了持久的产品骨架
- 后续 probe 工作可以接入一个真实系统，而不是一次性原型

### 16.4 里程碑 2：attach 与 observer 双入口

状态：

- 部分完成
- attach control path 已存在，并具备结构化 session state
- `Codex` / `opencode` observer 主干已接入
- observer-first 的统一产品面仍在收口

目标：

- 证明运行中的目标可以通过最合适的 source 接入：
  - Node / Electron 目标走 attach
  - 官方或准官方运行时走 observer

交付物：

- attach 路径：插桩后端适配器、bootstrap probe、heartbeat 与 detach 控制
- observer 路径：source 发现、握手、事件归一化与 artifact 落盘
- 通过 UI 或 CLI 控制路径完成 source 选择与接入动作

退出标准：

- 在开发机上，至少有一类 attach 目标可以稳定接入
- 至少有一类 observer 目标可以稳定输出高层事件
- 产品中能看到 source、session 和 failure 状态

这是决定成败的接入面验证点。

### 16.5 里程碑 3：Payload 捕获管线

状态：

- 未开始

目标：

- 从实时目标中捕获真正面向模型的请求与响应事实

交付物：

- 支持 `fetch` hook
- 支持 `undici` hook
- 提供 `http` 与 `https` 兜底支持
- 请求事件发射
- 响应事件发射
- 在需要时把原始 payload 持久化到 artifacts
- 把规范化事件写入 SQLite

退出标准：

- 至少能在存储中看到一个真实 provider 请求
- 请求与响应可以通过 request id 或等效键关联
- 在目标继续运行时，也能检查已捕获的 payload

这是产品第一次真正变得有意义。

### 16.6 里程碑 4：本地可观测性控制台

状态：

- 未开始

目标：

- 让捕获到的数据能以产品化方式被检查，而不只是调试转储
- 让 attach 与 observer 结果以统一 source 视图进入同一套控制台

交付物：

- source 列表页
- 实时会话时间线
- request / event 列表视图
- 统一 inspector 详情视图
- 如 provider、model、duration、stream 和 tool count 等基础元数据标签
- probe / observer 健康状态与错误可见性

退出标准：

- 用户可以附加到一个进程，并在不直接接触数据库的情况下检查捕获到的请求

这个里程碑把捕获基础设施转化成真正的可观测性体验。

### 16.7 里程碑 5：会话关联与搜索

状态：

- 未开始

目标：

- 让产品在“多于一个请求”的情况下依然可用

交付物：

- session 分组逻辑
- 可搜索的请求与响应内容
- 按 app、model、provider 和时间范围过滤
- 用于快速浏览的摘要预览
- 事件排序与时间线重建改进

退出标准：

- 用户可以从过去捕获的活动中找到某个具体请求，而不需要手动记住时间戳

这是产品从 demo 迈向工具的一步。

### 16.8 里程碑 6：加固与兼容性打磨

状态：

- 未开始

目标：

- 稳定第一版具有公开质量的产品切片

交付物：

- 已测试目标的兼容性矩阵
- 更清晰的 attach 失败分类
- payload 大小边界
- 对突发式流捕获的背压处理
- 基础数据保留控制
- 打包与本地安装工作流

退出标准：

- 工具在一小组经过测试的真实 Node 或 Electron AI 目标上足够可靠
- 已知不支持的场景被明确说明

这个里程碑同样关乎诚实与信任，而不仅仅是工程质量。

### 16.9 推荐的首批支持目标集

V1 一开始不应宣称广泛兼容。

更好的策略是：

- 选择一小组有代表性的 Node 或 Electron AI 工具
- 先把它们做好
- 明确公开兼容性边界

好的初始目标类别：

- 一个基于 Node 的 AI CLI
- 如果可行，一个基于 Electron 的桌面 AI 应用
- 一个用于重复测试的可控内部或演示目标

这样可以避免在模糊生态上盲目开发。

### 16.10 建议的内部开发顺序

在上述里程碑内部，更务实的工程顺序应是：

1. 事件模型与存储 bootstrap
2. 进程发现与 attach 控制路径
3. 插桩后端适配器
4. bootstrap probe 与 heartbeat
5. `fetch` 捕获
6. `undici` 捕获
7. 请求与响应持久化
8. 用于请求检查的本地 UI
9. session 分组与搜索
10. 兼容性与加固

这一顺序最大化了尽早看到真实 payload 的概率。

### 16.11 有意延后的事项

在 V1 捕获能力跑通之前，团队应明确延后以下内容：

- 深入解释为什么某个 skill 没有触发
- 跨 session 的自动 prompt diff
- 跨机器同步
- 云存储
- 多用户协作
- 对非 Node runtime 的支持
- 复杂仪表盘

这些都很有吸引力，但不应与“拿到第一批真实信号”竞争资源。

### 16.12 MVP 演示场景

MVP 应具备一个可重复演示的场景：

1. 启动可观测性工具
2. 选择一个正在运行的受支持 AI 应用
3. 成功附加
4. 在目标应用中发送一个新的 prompt
5. 看着一个请求出现在会话时间线中
6. 打开该请求，检查发送给模型的精确 payload

如果这条链路能顺畅跑通，项目就已经从想法跨入产品阶段。

### 16.13 对 V1 的团队心态建议

第一版应通过以下问题来评估：

- 我们是否从一个正在运行的应用中捕获到了真实 payload
- 我们是否在不强制重启的情况下做到这点
- 用户能否舒适地检查结果
- 存储下来的事实能否支撑未来分析

如果答案是肯定的，那么即便许多后续可观测性特性还缺失，V1 仍然走在正确轨道上。

## 19. 设计九：隐私、脱敏与安全边界

这个产品会捕获 AI 工作流中一些最敏感的数据。

根据被观测应用的不同，捕获到的 payload 可能包括：

- 专有 system prompts
- 内部工具或 skill 定义
- 源代码
- 客户数据
- 被粘贴进 prompt 的密钥
- 文件路径与本地环境细节

因此，隐私与安全不是次要考虑，而是产品设计的核心约束。

### 19.1 隐私立场

V1 应采用严格的 local-first 隐私立场。

默认规则：

- 数据留在本机
- 无云同步
- 无隐藏导出
- 不发送包含捕获 payload 内容的远程遥测
- 不向第三方服务做后台共享

这不仅更安全，也会成为产品的重要信任优势。

### 19.2 信任边界

系统应定义三个明确的信任区：

1. 被观测进程
2. 本地可观测性 host
3. 本地存储与 UI

重要规则：

- host 可能从被观测进程接收到敏感内容，但它只应在用户显式的本地操作下暴露这些内容

这意味着，捕获数据不应散落进日志、崩溃转储、分析系统或不受控的调试输出中。

### 19.3 数据分类

V1 应将捕获数据视为属于不同敏感等级。

建议分类：

- `public_metadata`
- `sensitive_prompt_content`
- `sensitive_response_content`
- `secret_like_material`
- `diagnostic_internal`

示例：

- provider 名通常属于 `public_metadata`
- system prompt 文本通常属于 `sensitive_prompt_content`
- API key 或类似 token 的字符串属于 `secret_like_material`
- hook 安装诊断属于 `diagnostic_internal`

这种分类能在不重构产品的情况下支持未来策略。

### 19.4 原始捕获策略

原始捕获对可信度与重处理都很重要，但必须谨慎处理。

V1 的原始捕获策略应为：

- 可以在本地存储原始请求体和响应体
- 原始 payload 默认应被清晰标记为敏感
- 原始 payload 不应在 UI 的每个地方自动渲染
- 原始 payload 应接受大小边界与后续脱敏处理

产品应假定原始捕获高风险但必要。

### 19.5 脱敏模式

V1 不需要完美脱敏，但应尽早定义模式。

推荐模式：

- `off`
- `display_only`
- `persist_redacted_copy`
- `persist_redacted_only`

含义：

- `off`：保留并展示完整本地数据
- `display_only`：原始数据保留在本地，但 UI 默认隐藏可能敏感的片段
- `persist_redacted_copy`：保留原始数据，同时保留一份派生的脱敏版本
- `persist_redacted_only`：对于选定 payload 类别，只持久化脱敏表示

V1 可以先从 `off` 和 `display_only` 开始，同时让存储模型为以后更强的模式做好准备。

### 19.6 类 Secret 检测

系统应规划在写入过程中进行轻量级的类 secret 检测。

示例：

- API key 模式
- bearer token
- 私钥头部
- 明显形似凭证的环境变量值

V1 规则：

- 除非用户选择的脱敏模式要求遮罩，否则检测只做标注，不应静默修改数据

这样既保留取证价值，也为更安全的默认行为做好准备。

### 19.7 UI 显示安全

UI 应避免意外暴露敏感内容。

推荐的 V1 显示保护：

- 默认折叠原始 payload 区域
- 对大段文本显式点击后才展开
- 对疑似 secret 内容做可视标记
- copy 动作应是有意触发，而非自动发生
- 预览字段应裁剪并有大小边界

这样既保持产品可用，又不会让每个界面都变成潜在泄漏面。

### 19.8 日志规则

应用日志绝不能成为捕获机密的旁路。

严格的 V1 规则：

- 不要把完整捕获 payload 写入常规应用日志
- 不要在崩溃摘要中包含 prompt 正文
- 默认不要在 debug 日志中打印敏感内容
- 诊断时使用 event id 和 artifact id，而不是正文转储

这是最容易意外破坏信任的地方之一，因此策略应足够严格。

### 19.9 导出边界

即使 V1 还不会立刻实现导出，现在也应先定义规则。

任何未来导出都必须：

- 显式触发
- 由用户主动发起
- 在完成前可预览
- 范围限定在所选 session 或 event 内

产品绝不能因为“数据已经在本地”就默认导出是无害的。

### 19.10 保留安全

敏感本地数据不应因疏忽而永久留存。

推荐的 V1 基础能力：

- 清晰展示存储占用
- 支持按 session、进程或时间范围删除
- 区分元数据删除与 artifact 删除
- 让完全清除行为可理解且可靠

这并不要求 V1 做激进的自动删除，但要求它对生命周期控制保持诚实。

### 19.11 Probe 安全边界

被注入的 probe 应遵守非常狭窄的契约。

规则：

- 只捕获事件模型所必需的内容
- 不尝试权限提升
- 不读取任意无关内存区域
- 不枚举与定义 hook 点无关的应用数据
- 在 host detach 时干净地禁用

这样能让 probe 始终围绕可观测性，而不会滑向通用进程抓取。

### 19.12 Host 安全边界

Rust host 应假定 probe 输入可能是畸形的，甚至是恶意的。

host 侧规则：

- 校验 probe 消息
- 限制 payload 大小
- 拒绝无效 schema 版本
- 将 attach 阶段元数据视为不可信输入
- 隔离 artifact 写入路径

即使用户观测的是自己的进程，host 也应保护自己免受畸形捕获流的影响。

### 19.13 未来的多用户与团队边界

产品未来可能扩展到团队工作流，但 V1 不应假设共享存储天然安全。

未来共享应被视为一个独立的信任模型，而不是本地模式的自然延伸。

这意味着 V1 应避免：

- 隐式的团队同步假设
- 面向服务端存储的捷径
- 默认认为所有捕获内容都可以安全复制的 schema

### 19.14 推荐的 V1 隐私默认值

推荐的 V1 默认设置是：

- 仅本地存储
- 日志中不包含 payload 内容
- UI 中默认折叠原始内容
- 在元数据中打上敏感性标记
- 支持删除 session 和 artifact
- 为后续可选脱敏模式做好准备

这些默认值在可观测性价值与用户信任之间取得了良好平衡。

### 19.15 产品定位收益

隐私控制不仅是防御性工程。

它也会强化产品叙事：

- 这个工具帮助用户检查真实的 AI 行为
- 它在本地完成这一切
- 它不会强迫用户把捕获到的 prompts 交给另一个服务

这种定位可以成为真正的差异化优势。
