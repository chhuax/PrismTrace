## 迭代拆分原则

这份路线图采用"独轮车 -> 自行车 -> 摩托车 -> 汽车"的切法，而不是按内部零件切分。

也就是说：

- 每个迭代都必须形成一个能运行、能演示、能验证价值的产品切片
- 每个迭代都应该回答"用户现在多获得了什么能力"
- 内部组件建设只能作为支撑，不应成为单独迭代目标

因此，下面的迭代定义都以"整体验证能力"为中心，而不是按轮子、车门、座椅这种内部模块来拆。

### 迭代 1：独轮车 - 候选目标发现器

状态：已完成

目标：

- 用户可以在本机上跑出"哪些 AI 相关进程值得看"

已交付：

- 结构化 `ProcessTarget`

对应 change：

- 对 Node / Electron / Unknown 做初步分类
- 为后续 attach 铺路

### 迭代 2：自行车 - 可附着目标检查器

状态：已完成

目标：

- 用户不只是"看到一堆进程"，而是能知道"哪些目标现在值得 attach"
- 在真正注入 probe 之前，先把 attach readiness 跑通

产品形态：

- 输入：候选进程列表
- 输出：可附着 / 不可附着 / 原因 / 风险提示

用户可见能力：

- 看到候选目标的 readiness 状态
- 看到失败分类和建议动作
- 明白下一步 attach 是否值得做

已交付：

- `attach-readiness` 主 spec
- 结构化 `AttachReadiness` 结果模型
- `supported / unsupported / permission_denied / unknown` 状态集
- host 侧 readiness service
- `--readiness` 本地入口和确定性测试

对应 change：

- [x] `add-attach-readiness`

### 迭代 3：电动自行车 - 可连接但暂不采集

状态：已完成

目标：

- 用户可以真正发起 attach，并看到连接状态
- 即使此时还没有完整 payload capture，也已经完成"连接到目标"的闭环

产品形态：

- 输入：一个 readiness 通过的目标
- 输出：attach / detach / attach state / 失败原因

用户可见能力：

- 选择目标并发起 attach
- 看到 attach 是否成功
- 看到 attach 生命周期和失败反馈
- 看到 probe 健康状态（已安装 hook 列表、失败 hook 列表）

已交付：

- `attach-controller` 主 spec
- attach session 生命周期、结构化失败结果和最小 probe bootstrap 领域模型
- host 侧 attach controller 和 scripted backend
- `--attach <pid>` 本地演示入口
- `add-probe-bootstrap` spec
- `InstrumentationRuntime` trait（可替换插桩后端适配器）
- `LiveAttachBackend`：真实 attach backend，等待 `BootstrapReport` 握手，detach 等待 `DetachAck`
- bootstrap probe（`probe/bootstrap.js`）：运行时检测、hook 骨架安装、stdout IPC、heartbeat、detach 处理
- `IpcListener`：面向行 JSON 读取，跳过非 IPC 行，heartbeat 超时检测
- `ProbeHealthStore`：probe session 状态管理
- `--detach` / `--attach-status` CLI 命令面
- 77 个确定性测试全部通过

已知限制（留待后续迭代）：

- 跨命令 session 持久化需要 daemon 架构
- heartbeat 持续监控需要长进程模型
- `NodeInstrumentationRuntime` 是占位符，真实插桩后端待实现

对应 change：

- [x] `add-attach-controller`
- [x] `add-probe-bootstrap`

### 迭代 4：摩托车 - 单目标请求采集器

## 迭代优先级建议

如果按"最快形成产品闭环"的目标排序，我建议这样推进：

1. 迭代 4：摩托车 - 单目标请求采集器（真实 payload capture）
2. 迭代 5：汽车 - 本地可观测性控制台
3. 迭代 6：越野车 - 会话重建与可见性增强
4. 迭代 7：自动驾驶辅助 - 分析与解释层

- 一个 change 只解决一个清晰能力边界
- 一个 change 最好能在 1 到 3 次实现 session 里完成

## 什么时候需要重切路线图

出现下面任一情况时，应该回头更新这份路线图：

- attach/instrumentation backend 选型发生变化
- 第一版产品 surface 不再是本地 host + Web UI
- 目标平台不再局限于 macOS
- 你决定先做 CLI-first 调试器，而不是 observability console
