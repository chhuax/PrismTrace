## 当前产品判断

PrismTrace 的方向已经从早期 `attach-first payload capture` 收敛为：

`本地优先的 AI coding agent 可观测与诊断工具。`

当前主线是：

- observer-first，而不是 attach-first
- 先保留真实运行时事实，再做分析解释
- 先打通 Codex / Claude Code / opencode 等 coding agent 的高层事件
- 以 session、event、capability projection 为后续分析底座
- 最终形成 `Skill Diagnostics for AI Coding Agents`

PrismTrace 不应被理解为：

- 通用抓包器
- 通用 benchmark / eval 平台
- 云端 observability SaaS
- 单一 AI CLI 的逆向工程工具

## 历史路径说明

早期路线按 discovery、attach、request capture、response capture、local console、session reconstruction 逐步推进。这条路径有历史价值，但不再代表下一阶段优先级。

后续不建议继续围绕“让 attach 更强”展开主路线。attach / probe 可以作为特定 source 的补充能力存在，但产品默认入口应是 observer source 和本地可观测控制台。

## 迭代原则

后续迭代按以下原则拆分：

- 每个迭代都要形成可运行、可演示、可验证的产品切片
- 架构建设必须服务用户可见能力
- raw facts、index、analysis、api、console 分层推进
- 不直接从 diagnostics 开始，先补齐可查询事实层
- 不为单个 source 过度定制 analysis 逻辑
- 保留 raw JSON，projection 只承担稳定查询和分析入口

## 总体路线

新的产品化路线分为六个阶段：

```text
0. 文档与方向收敛
1. Observability read model 拆分
2. Console 性能与本地体验修复
3. Capability / Skill Visibility
4. Capability Diff
5. Skill Diagnostics
6. Failure Case 与回归样本
```

## 当前最推荐的下一步

如果只开一个下一提案，当前最推荐：

`split-observability-read-model`

原因：

- 它直接解决当前 console 不流畅的根因之一
- 它把 source-specific 文件读取从 console 中拆出来
- 它为 skill visibility 和 skill diagnostics 提供共同数据底座
- 不先做 read model，后续 diagnostics 会继续耦合在 console / source parser 里

## Change 优先级建议

建议按以下顺序推进：

1. `split-observability-read-model`
2. `improve-console-performance`
3. `add-skill-visibility`
4. `add-capability-diff`
5. `add-skill-diagnostics`
6. `add-failure-case-capture`

## 何时更新路线图

出现以下情况时，应更新本文档：

- 当前产品方向不再是 `Skill Diagnostics for AI Coding Agents`
- 本地 Web console 不再是主要产品 surface
- source 类型显著扩展，超出 Codex / Claude Code / opencode
- analysis 层能力从本地诊断转向云端服务
- `docs/产品方案.md` 或 `docs/技术方案.md` 发生重大调整
