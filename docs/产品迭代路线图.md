日期：2026-04-19
更新：2026-04-26
状态：进行中

## 目标

把现有设计方案拆成一条可以持续推进的研发路线，明确：

- 当前已经完成了什么
- 后续应该拆成多少个迭代
- 每个迭代交付什么能力
- 每个迭代建议对应哪些 OpenSpec change

这份文档的用途不是替代单个 change 的 proposal/design/tasks，而是作为"上层版本路线图"，帮助后续按优先级开提案和排迭代。

- 当前产品主线先解决“官方运行时高层事件采集”
- `Codex` 是第一条被真正打通并进入产品面的 observer source
- 控制台与后续分析层需要以统一 `source / event / session / inspector` 模型消费 `Codex`，并为 `opencode` 预留同壳接入

## 迭代拆分原则

这份路线图采用"独轮车 -> 自行车 -> 摩托车 -> 汽车"的切法，而不是按内部零件切分。

也就是说：

- 每个迭代都必须形成一个能运行、能演示、能验证价值的产品切片
- 每个迭代都应该回答"用户现在多获得了什么能力"
- 内部组件建设只能作为支撑，不应成为单独迭代目标

因此，下面的迭代定义都以"整体验证能力"为中心，而不是按轮子、车门、座椅这种内部模块来拆。

## 迭代优先级建议

如果按"最快形成产品闭环"的目标排序，我建议这样推进：

1. 迭代 4：摩托车 - 单目标请求采集器（真实 payload capture）
2. 迭代 5：汽车 - 本地可观测性控制台
3. 迭代 6：越野车 - 会话重建与可见性增强
4. 迭代 7：自动驾驶辅助 - 分析与解释层

## 当前最推荐的下一步

如果只开一个下一提案，我当前最推荐：

- 先完成 `add-codex-app-server-observer`

原因：

- 路线已经从 attach-first 调整成 multi-source observer-first，但产品面还没完全跟上
- 先把 `Codex` 与 `opencode` 的 observer 能力接进统一控制台，才能证明新的产品主线成立
- 在 unified source / event / session 这层收稳之后，再进入 `prompt diff`、`failure attribution`、`skill diagnostics` 会更顺

## 什么时候需要重切路线图

出现下面任一情况时，应该回头更新这份路线图：

- attach/instrumentation backend 选型发生变化
- observer source 类型显著扩展，超过 `Codex / opencode` 当前抽象
- 第一版产品 surface 不再是本地 host + Web UI
- 目标平台不再局限于 macOS
- 你决定先做 CLI-first 调试器，而不是 observability console
