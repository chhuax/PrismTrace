# PrismTrace Architecture Refresh

日期：2026-04-30
状态：全局架构方案

## 概览

PrismTrace 的主方向仍然成立：本地优先、observer-first、保留 raw facts，并在事实层之上构建 session timeline、prompt / tool / skill analysis。

当前需要调整的不是产品方向，而是工程架构。现有实现已经从 bootstrap 阶段进入产品化阶段，但代码职责仍集中在 `prismtrace-host` 和 `console` 模块里，导致性能、复用和后续分析能力都会被牵制。

本方案是全局架构方案，不绑定某个 story。它为后续 `Skill Diagnostics for AI Coding Agents`、prompt diff、tool visibility diff、session replay 等能力提供共同底座。

## 当前问题

### Host 过重

`prismtrace-host` 当前同时承担：

- CLI orchestration
- observer source 接入
- artifact 写入
- session reconstruction
- console API
- 静态资源服务

早期这样做能快速闭环，但继续增长会让每个新能力都挤进同一个运行时层。

### Console 模块承担了数据层职责

`console/observer.rs` 同时负责：

- 读取 Codex rollout JSONL
- 读取 observer event artifacts
- 归一化 session summary
- 生成 session detail
- 生成 request detail payload
- 服务 console API

这会让 console 变成事实上的数据访问层，后续 analysis 和 diagnostics 很难复用。

### 缺少索引层

PrismTrace 已经有 artifacts，但缺少稳定 index。现在 console API 可能在请求时扫描 `~/.codex/sessions`，再解析 JSONL、过滤、排序、截断。

当本机 Codex session 文件达到几十个、上百 MB 后，`/api/sessions` 和 `/api/sessions/:id` 会明显变慢。

### 本地 HTTP server 只适合 demo

当前同步单线程 server 会让慢 API 阻塞静态资源和后续请求。用户体感上会表现为页面加载慢、点击卡、资源偶尔排队。

### Analysis 没有独立位置

`prompt diff`、`tool visibility diff`、`skill visibility`、`skill diagnostics` 都应该属于 analysis 层，而不是散落在 source parser 或 console renderer 中。

## 目标架构

建议将 PrismTrace 收敛为以下层次：

```text
sources
  codex app server observer
  claude transcript observer
  opencode server observer
  future attach/probe source

ingest
  normalize source events
  preserve raw_json
  write artifacts

index
  build session index
  build event index
  build capability index
  cache summaries

analysis
  prompt projection
  prompt diff
  tool visibility diff
  skill visibility
  skill diagnostics

api
  read-only local HTTP API
  paginated sessions/events/details
  diagnostics endpoints

console
  local UI
  renders API payloads
  avoids direct file scanning assumptions
```

核心边界：

`Source 负责观察，Ingest 负责落事实，Index 负责可查询，Analysis 负责解释，API 负责暴露本地读模型，Console 负责展示。`

## 模块演进

### 第一阶段：先在 host 内拆清边界

短期不必立刻拆 crate，可以先在 `prismtrace-host` 内形成清晰模块：

```text
crates/prismtrace-host/src/sources/
crates/prismtrace-host/src/ingest/
crates/prismtrace-host/src/index/
crates/prismtrace-host/src/analysis/
crates/prismtrace-host/src/api/
crates/prismtrace-host/src/console/
```

迁移方向：

- `codex_observer.rs`、`claude_observer.rs`、`opencode_observer.rs` 进入 `sources`
- observer artifact writer 和 raw event normalization 进入 `ingest`
- Codex rollout session 扫描、session summary、event lookup 进入 `index`
- prompt diff、tool diff、skill visibility、diagnostics 进入 `analysis`
- HTTP route 和 JSON payload 进入 `api`
- HTML/CSS/JS 静态资源、页面 shell、前端代码留在 `console`

### 第二阶段：边界稳定后拆 crate

后续可演进为：

```text
prismtrace-core
prismtrace-storage
prismtrace-sources
prismtrace-index
prismtrace-analysis
prismtrace-api
prismtrace-host
```

其中 `prismtrace-host` 只保留 CLI 和进程生命周期编排。

## 数据模型

后续全局分析能力至少需要三个 projection。

### Session Projection

用于回答有哪些会话、来自哪个 source、时间范围是什么。

建议字段：

- `session_id`
- `source_kind`
- `source_label`
- `title`
- `cwd`
- `started_at_ms`
- `completed_at_ms`
- `event_count`
- `artifact_refs`

### Event Projection

用于回答某个 session 中发生了哪些可观察事件。

建议字段：

- `event_id`
- `session_id`
- `source_kind`
- `event_kind`
- `role`
- `summary`
- `occurred_at_ms`
- `correlation_keys`
- `raw_ref`

### Capability Projection

用于回答某一时刻 agent 可见哪些 skills、tools、plugins、apps。

建议字段：

- `capability_id`
- `session_id`
- `event_id`
- `source_kind`
- `capability_type`
- `capability_name`
- `visibility_stage`
- `observed_at_ms`
- `raw_ref`

Projection 不替代 raw artifacts，只提供稳定查询和分析入口。

## Storage / Index 策略

短期推荐：

```text
artifacts/
  raw facts and observer events

index/
  derived summaries and projections
```

实现原则：

- 首次读取 source artifacts 时构建 index
- index 记录源文件 path、mtime、size
- 源文件未变化时直接读取 index
- 源文件变化时只重建受影响部分

进入 SQLite 的条件：

- session 文件扫描持续成为性能瓶颈
- 需要复杂筛选、分页、跨 session 查询
- diagnostics 需要稳定引用历史样本

## API 调整

本地 API 应从“返回全部”转为分页和按需读取：

```text
GET /api/sessions?source=codex&limit=30&cursor=...
GET /api/sessions/:session_id
GET /api/sessions/:session_id/events?limit=100&cursor=...
GET /api/events/:event_id
GET /api/sessions/:session_id/capabilities
GET /api/sessions/:session_id/diff?baseline=previous
POST /api/diagnostics/skill
```

原则：

- summary API 不返回大体积 raw payload
- detail API 按需读取 raw artifact
- timeline 默认分页
- diagnostics API 只基于本地事实运行

## Console 调整

Console 可以继续是本地 Web UI，但需要避免当前不流畅来源：

- 移除 Tailwind CDN，改为本地静态 CSS 或构建期 CSS
- 字体和 icon 本地化
- session list 和 timeline 做分页或虚拟列表
- checkbox、筛选、展开详情避免全量重建整条 timeline
- detail panel 按需加载 raw payload
- capability / diagnostics 作为明确视图，而不是混在 request detail 里

前端框架不是第一优先级。应先稳定 API / index，再决定继续原生 HTML/JS，还是切到 Vite + Svelte / Solid / React。

## Skill Diagnostics 的架构位置

`Skill Diagnostics for AI Coding Agents` 不应直接读取 Codex、Claude、opencode 原始文件。

正确路径：

```text
source raw events
  -> ingest normalized events
  -> index session/event/capability projections
  -> analysis diagnostics
  -> api diagnostics result
  -> console diagnostics panel
```

这样 diagnostics 可以跨 source 复用，也能在事实不足时明确返回 `partial` 或 `unavailable`。

## 推荐实施顺序

1. 抽出 Codex rollout index，避免 `/api/sessions` 每次扫描 `~/.codex/sessions`。
2. 并发化本地 HTTP server，避免慢 API 阻塞静态资源。
3. API 改为分页和 detail 按需读取。
4. 新增 capability projection，支撑 skill / tool / plugin / app visibility。
5. 新增 analysis 模块，承载 prompt diff、tool diff、skill diff、skill diagnostics。
6. 优化 console 体验，包括本地资源化、减少全量重绘、timeline 分页或虚拟化。

## 验收标准

- `/api/sessions` 不再每次全量扫描 Codex session 目录
- 静态资源请求不被慢 API 长时间阻塞
- session、event、capability 三类 projection 有清晰模型
- console 数据读取逻辑可被 analysis 复用
- Skill Diagnostics 可以基于 projection 实现，而不是耦合某个 source 文件格式
