# Skill Diagnostics for AI Coding Agents

日期：2026-04-30
状态：全局产品方案

## 概览

`Skill Diagnostics for AI Coding Agents` 是 PrismTrace 的全局产品方向之一。

它面向使用 Codex、Claude Code、opencode 或团队内部 coding agent 的软件工程师，帮助他们理解真实会话中：

- agent 看到了哪些 skills、tools、plugins、apps
- 某次请求最终携带了哪些能力描述
- prompt、tool visibility、skill visibility 在相邻 turn 之间发生了什么变化
- 为什么用户预期某个 skill 应该触发，但实际没有触发

这个方向不是通用离线 eval，也不是云端监控平台。它延续 PrismTrace 的本地可观测性定位：

`观察真实运行时事实，再基于事实做诊断。`

## 产品定位

英文定位：

`PrismTrace helps developers understand which skills and tools were visible to an AI coding agent, what changed between turns, and why a skill may not have triggered.`

中文定位：

`PrismTrace 帮助开发者理解 AI 编程助手在真实会话中看到了哪些 skill / tool，哪些能力进入了请求，以及某个预期能力为什么可能没有触发。`

核心价值：

- 让开发者不再只凭感觉猜 agent 为什么没用某个 skill
- 把隐藏在 prompt、tool schema、observer event、transcript 里的事实组织出来
- 为 prompt / skill / workflow 回归测试提供真实样本基础
- 让 AI coding agent 调试从“看最终回答”前移到“看能力可见性和上下文变化”

## 目标用户

第一优先级用户：

- 使用 AI coding agent 的软件工程师
- 维护 Codex / Claude Code / opencode workflow 的开发者
- 编写或维护 custom skills / prompts / tools 的开发者

第二优先级用户：

- 团队内部 AI platform / developer productivity 工程师
- 想比较不同 agent 行为差异的工具作者
- 想做 agent workflow 回归测试的人

非目标用户：

- 只想跑标准模型 benchmark 的 ML researcher
- 只想监控线上客服机器人的业务运营人员
- 需要云端多租户观测平台的企业管理员

## 用户痛点

### 不知道 skill 有没有被看见

开发者以为某个 skill 已经配置好了，但 agent 实际请求中可能没有携带相关描述，或者 observer source 没有报告对应 capability。

用户问题：

`这个 skill 到底有没有进入本轮上下文？`

### 不知道 skill 为什么没触发

用户输入看起来符合 skill 描述，但 agent 没有使用它。

可能原因：

- skill 不在可见能力列表中
- prompt 中的触发条件被其他指令稀释
- tool schema 存在但 tool choice 未倾向使用
- session 中缺少必要上下文
- approval、hook 或 source 限制导致路径中断

用户问题：

`是 skill 没注册、没进请求，还是模型选择没用？`

### 改 prompt / skill 后不知道行为变化

开发者修改 skill 文档、系统提示、agent workflow 后，只能靠人工对比最终回答。

用户问题：

`这次改动到底改变了哪些能力可见性和上下文？`

### 真实失败样本无法沉淀

一次失败会话里可能有有价值的上下文、工具调用、skill snapshot，但它们散落在 transcript、request payload 和 observer events 中。

用户问题：

`我怎么把这次失败变成后续可复盘、可回归的样本？`

## 产品原则

- 本地优先：不上传会话、prompt、payload、transcript
- 事实优先：先展示证据，再给出可能解释
- 可追溯：每条诊断结论都能回到 raw event 或 artifact
- source-aware：允许 Codex、Claude、opencode 的事实粒度不同
- 不过度承诺：事实不足时明确显示 `partial` 或 `unavailable`
- 面向开发者：输出应能帮助修 prompt、修 skill、修 workflow

## 核心对象

### Skill

可复用的 agent 能力描述。它可能来自 Codex skill、Claude Code skill / command、Cursor rule，或团队内部 agent workflow 模块。

### Tool

模型可调用或 agent 可执行的工具，包括 function call、MCP tool、本地 command wrapper 等。

### Plugin / App

更高层的能力容器，可能提供多个 tools 或 skills。

### Capability Snapshot

某个时刻 agent 可见能力的快照。它可以包含 skills、tools、plugins、apps，也可以只包含其中一部分。

### Diagnostic Evidence

诊断所引用的事实证据，例如 capability snapshot、request-embedded tool visibility、prompt diff、timeline event、tool call、approval、hook、error event、raw JSON artifact。

## 产品形态

### Skill Visibility

回答：

`当前会话或当前请求中，哪些 skills / tools / plugins / apps 是可见的？`

体验：

- 在 session detail 中看到 `Capabilities` 区块
- 按 skill、tool、plugin、app 分组
- 每个 capability 显示名称、来源、visibility stage、出现次数、最近观察时间
- 支持打开 raw JSON 查看原始证据

第一版只展示事实，不解释原因。

### Skill Diff

回答：

`这次请求相比上一轮，能力面发生了什么变化？`

体验：

- 在 timeline 中选择一个 turn
- 看到与上一 turn 的 capability diff
- 展示新增、移除、数量变化、名称变化
- 将 prompt diff、tool visibility diff、skill visibility diff 放在同一个比较视图中

第一版默认只比较同一 session 内相邻 turn。

### Skill Diagnostics

回答：

`我预期某个 skill 应该触发，但它没有触发，可能原因是什么？`

体验：

- 用户输入 expected skill 名称，或从 capability 列表中选择
- PrismTrace 生成诊断摘要
- 摘要分为 observed facts、missing evidence、likely explanations、next checks
- 每个结论都链接到对应 event、request 或 artifact

示例：

```text
Expected skill: gh-address-comments

Observed facts:
- No skill capability named gh-address-comments was observed in this session.
- The request carried 18 tools, but none matched GitHub PR review workflows.
- The user request did mention "review comments".

Likely explanations:
- The skill was not visible to the agent for this turn.
- The agent may have treated the request as a general code task instead of a GitHub review workflow.

Next checks:
- Verify skill discovery configuration.
- Compare this turn with a known session where the skill was visible.
```

## MVP

推荐 MVP：

`Skill Visibility for Codex observer sessions`

第一版只做可见性，不做完整 diagnostics。

目标：

- 从已有 observer events 中识别 skill、plugin、app、tool
- 从 request payload 中复用 request-embedded tool visibility
- 生成 session-scoped capability snapshot
- 在 console 中展示 capability list
- 保留 raw JSON 证据入口

非目标：

- 不解释为什么某个 skill 没触发
- 不做跨 session 回归测试
- 不做自动 benchmark
- 不支持云端同步
- 不要求所有 source 都有同等 capability 粒度

验收标准：

- 对至少一个真实 Codex observer session，能展示可见 capabilities
- 对没有 capability 事实的 session，显示明确空态
- 每个 capability 能追溯到 source、session、event 或 request artifact
- UI 不阻塞现有 request / response / timeline 浏览
- 不改变现有 raw artifact 格式

## 版本路径

### V0.2：Skill Diff

- 同一 session 相邻 turn diff
- 新增 / 移除 / 变化 capability 展示
- 与 prompt diff、tool visibility diff 合并展示

### V0.3：Expected Skill Diagnostics

- 用户指定 expected skill
- 检查是否可见、是否进入请求、是否有相关 tool call
- 输出基于事实的诊断摘要

### V0.4：Failure Case Capture

- 将一次诊断结果保存为 failure case
- 记录输入、上下文摘要、expected skill、证据链接
- 为未来 regression eval 做准备

### V0.5：Cross-Agent Comparison

- 比较 Codex / Claude Code / opencode 在同类任务中的 capability visibility
- 识别不同 source 的可观测性缺口

## Console 信息架构

### Session Overview

展示 source、project / cwd、event count、capability count、tool call count。

### Timeline

每个 turn 可显示是否有 capability snapshot、tool visibility、prompt diff、diagnostic note。

### Inspector

选择某个 turn 或 event 后展示 raw event、prompt projection、tool visibility、skill visibility、related diagnostics。

### Diagnostics Panel

用于输入 expected skill 或选择 capability 后生成诊断。第一版可以是只读面板，不需要聊天式交互。

## 与架构方案的关系

本产品方案依赖全局架构方案中的：

- session projection
- event projection
- capability projection
- indexed local query
- analysis layer
- paginated local API

推荐数据路径：

```text
source raw events
  -> ingest normalized events
  -> index session/event/capability projections
  -> analysis diagnostics
  -> api payload
  -> console view
```

## Open Questions

- 第一版 capability snapshot 应按 session 聚合，还是按 turn/request 保留？
- Codex 的 skill / plugin / app 事件语义是否足以支撑 MVP？
- Claude Code transcript 中是否能稳定看到 skill 或 command 信息？
- UI 上 skill 和 tool 是否需要严格区分，还是统称 capability？
- Expected skill 是用户手动输入，还是从已知 skill registry 中选择？
- failure case 是否应该成为独立 artifact 类型？
