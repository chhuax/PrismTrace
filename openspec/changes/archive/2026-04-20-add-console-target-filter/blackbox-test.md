# 黑盒测试说明

## 测试目标

- 验证本地控制台在启动时支持限定监控目标，而不是始终回退到全局 discover 视图
- 验证带过滤条件的控制台首页只展示匹配目标的 targets / activity / requests / observability health
- 验证控制台会明确展示当前过滤条件
- 验证无匹配目标时仍能正常打开控制台，并返回与过滤上下文一致的空态结果
- 验证未传过滤参数时，既有 `add-local-console` 默认行为不回退

## 测试范围

- `prismtrace-host -- --console` 的过滤启动路径
- 控制台首页的外部可见行为：Targets / Activity / Requests / Request Detail / Observability Health
- 过滤上下文在首页与局部刷新 payload 中的一致性
- 无匹配目标、默认未过滤和过滤命中三类行为分支

## 前置条件

- 本机可运行 `prismtrace-host`
- 本地环境中至少能构造出一个匹配目标和一个非匹配目标；或使用受控测试数据/夹具模拟该状态
- 浏览器可访问本地控制台地址
- 若做黑盒自动化，测试环境需能稳定控制控制台启动参数与可见目标集合

## 操作约束

- 本次验证只关注“控制台监控范围过滤”的外部行为，不验证底层 attach / detach 能力是否被改变
- 本次验证不覆盖 UI 内动态编辑过滤条件，因为这不在本 change 范围内
- 未传过滤参数时，应继续视为默认全局控制台模式，而不是一种“空过滤”特例

## 核心场景

### 1. 使用单个关键字启动过滤控制台

- 场景类型：成功
- 输入：使用单个目标关键字启动控制台，例如 `prismtrace-host -- --console <target-filter-flag> opencode`
- 关注点：
  - 控制台成功启动并可访问
  - 页面明确显示当前过滤条件为 `opencode`
  - `Targets` 区域只展示命中 `opencode` 的目标
- 预期：
  - 控制台不会回退为默认全局视图
  - 页面中不存在未匹配关键字的目标条目

### 2. 使用多个关键字启动过滤控制台

- 场景类型：成功
- 输入：使用多个目标关键字启动控制台，例如 `prismtrace-host -- --console <target-filter-flag> opencode <target-filter-flag> codex`
- 关注点：
  - 页面明确显示多个过滤条件摘要
  - 命中任一关键字的目标都可进入视图
  - 未匹配目标不会出现在列表中
- 预期：
  - 控制台以“任一关键字命中”语义工作
  - 过滤后的视图覆盖多个目标，但范围仍然受限

### 3. 活动与请求视图遵守过滤范围

- 场景类型：成功
- 输入：系统同时存在匹配目标与非匹配目标产生的 activity / request 记录，并以过滤关键字启动控制台
- 关注点：
  - `Activity` 只显示匹配目标的活动
  - `Requests` 只显示匹配目标的请求摘要
  - `Request Detail` 不暴露未匹配目标的 request 内容
- 预期：
  - 控制台所有首页聚合区域都使用同一过滤范围
  - 不存在“Targets 已过滤但 Activity/Requests 仍是全局”的不一致表现

### 4. 健康信息遵守过滤范围

- 场景类型：成功
- 输入：系统中同时存在匹配目标与非匹配目标的 probe/错误信息，并以过滤关键字启动控制台
- 关注点：
  - `Observability Health` 只反映匹配目标范围内的 probe 与错误摘要
  - 页面能说明这是一个带过滤上下文的健康视图
- 预期：
  - 健康摘要不会泄漏未匹配目标的信息

### 5. 过滤条件无匹配目标

- 场景类型：失败
- 输入：使用一个当前没有命中任何目标的过滤关键字启动控制台
- 关注点：
  - 控制台仍能正常打开
  - 页面仍显示当前过滤条件
  - `Targets` / `Activity` / `Requests` 返回与过滤上下文一致的空态
- 预期：
  - 不会回退到全局 discover 结果
  - 不会只显示模糊空白页或通用异常页

### 6. 未传过滤参数的默认控制台行为保持不变

- 场景类型：回归
- 输入：以既有方式启动控制台，不传任何过滤参数
- 关注点：
  - 控制台可正常启动
  - 页面不出现误导性的过滤上下文说明
  - 原有 `add-local-console` 的 targets / activity / requests / health 行为继续可用
- 预期：
  - 新增过滤能力不改变默认启动路径的外部行为

## 通过标准

- 带过滤参数启动时，控制台始终表现为“受限范围视图”而不是全局视图
- 首页可以明确看到当前过滤条件
- Targets / Activity / Requests / Health 的可见结果彼此一致，不出现部分区域未过滤的情况
- 无匹配目标时控制台正常打开，并返回明确空态
- 未传过滤参数时，默认控制台行为不退化

## 回归重点

- `add-local-console` 已有控制台入口与首页主区域仍然成立
- 未传过滤参数时的 API surface 与页面语义不被误改
- 过滤能力不会被误实现为 attach / detach 范围控制

## 自动化验证对应

- `crates/prismtrace-host/src/main.rs` 相关参数解析测试
  - 覆盖单个关键字、多关键字、未传过滤参数路径
- `crates/prismtrace-host/src/console.rs` 相关过滤聚合测试
  - 覆盖 targets / activity / requests / health 的过滤裁剪
  - 覆盖过滤上下文展示与无匹配空态
- 控制台集成测试 / 黑盒自动化
  - 覆盖过滤启动、页面上下文可见性、默认回归路径

## 测试环境待补充项

- 需要补充最终采用的过滤参数 flag 名称，并把黑盒命令示例替换为真实 CLI 形式
- 需要补充一组稳定的受控目标夹具，避免黑盒验证依赖开发机上的随机进程状态
