## 概览

`add-console-target-filter` 的目标是在已经具备最小浏览闭环的本地控制台上，增加一个“启动时限定监控范围”的能力。用户在真实环境里往往只关心一个或少数几个 AI 应用，例如 `opencode`、`codex`，但当前控制台默认展示全局 discover 结果，会把大量无关 Node / Electron 目标混进来，显著削弱控制台的可读性。

本次方案选择在控制台启动阶段引入目标过滤条件，并让首页聚合结果在同一过滤上下文下工作：Targets、Activity、Requests 与 Observability Health 全部只反映匹配目标的范围。这样可以在不改变底层 attach / detach 能力边界的前提下，把控制台从“全局扫描视图”推进到“用户可指定观察对象的局部监控视图”。

## 背景

`add-local-console` 已经为 PrismTrace 提供了：

- `prismtrace-host -- --console` 的本地入口
- target 列表、活动时间线、request 摘要列表与基础 request 详情
- 基础 observability health 与错误提示

但它仍然默认采用全局 discover 视图。对于真实开发者来说，这会带来两个问题：

- 打开控制台时会看到过多无关目标，难以快速定位当前想观察的 AI 应用
- 活动与 request 列表会混入其他本地 Node / Electron 目标的数据，降低可解释性

因此，下一步合理的收敛不是直接做复杂 UI 过滤器，而是先补上“启动时过滤范围”这一层，让控制台从一开始就进入正确的上下文。

当前约束：

- 仍是本地单机、macOS only
- 仍由 `prismtrace-host` 负责控制台启动与聚合，不引入新的独立控制面
- 过滤只约束控制台的监控与展示范围，不改变底层 attach / detach 行为
- 本次不扩展为 UI 内动态编辑过滤器或复杂检索语言

## 目标 / 非目标

**目标：**

- 在 `prismtrace-host -- --console` 路径上增加一个或多个目标关键字过滤参数
- 将过滤条件统一应用到 target、activity、request、health 的控制台聚合结果
- 在控制台首页与相关 payload 中明确显示当前过滤条件
- 当过滤条件没有命中任何目标时，控制台仍保持可访问并给出解释性空态

**非目标：**

- 改变底层 attach / detach / request capture 的实际作用范围
- 在控制台 UI 中提供运行时增删过滤条件的交互
- 支持正则表达式、复杂逻辑表达式或全文搜索式过滤语法
- 把本 change 扩展成完整的 request inspector 或高级查询能力

## 架构与方案概览

整体方案分三层：

1. **CLI 参数层**
   - 在 `main.rs` 中为 `--console` 路径增加目标过滤参数解析
   - 支持未传过滤参数、单关键字、多关键字三类输入

2. **控制台过滤配置层**
   - 在 `console.rs` 中增加控制台过滤配置结构
   - 提供“命中任一关键字”的匹配语义
   - 过滤目标依据优先使用 display name、进程名或等价 target 标识

3. **过滤后聚合与渲染层**
   - 将过滤条件接入 target / activity / request / health 的聚合过程
   - 页面与 `/api/*` payload 同步携带过滤上下文摘要
   - 无匹配时返回一致的空态，而不是偷偷回退为全局视图

## 关键设计决策

### 决策 1：过滤能力放在控制台启动参数，而不是先做 UI 内交互

- 背景：当前控制台仍是 bootstrap 阶段，最重要的是先建立稳定、可解释的“受限范围视图”。
- 方案：启动时传入过滤参数，控制台在整个生命周期内使用该范围。
- 备选方案：在 UI 中增加可编辑筛选栏，运行时切换过滤条件。
- 取舍：启动参数实现路径更短、测试更稳定，也避免在本轮引入新的前端状态复杂度；代价是用户暂时不能在页面内动态切换范围，但这符合当前迭代边界。

### 决策 2：过滤作用于控制台聚合结果，而不是底层 attach 范围

- 背景：用户需要的是“只看我关心的目标”，而不是重写底层 attach controller 的行为边界。
- 方案：过滤只裁剪控制台读模型和页面可见结果。
- 备选方案：让过滤参数直接影响 attach / detach / request capture 后台行为。
- 取舍：只过滤展示层更安全，也不会把本轮需求扩展成运行时控制策略改造；代价是底层仍可能采集全局事实，但当前产品目标是先解决“可见性上下文”。

### 决策 3：无匹配时维持过滤语义，不回退为全局结果

- 背景：如果过滤条件未命中时偷偷展示全局结果，用户会误判当前监控范围。
- 方案：控制台继续打开，但所有首页区域进入与过滤条件一致的空态。
- 备选方案：无匹配时自动回退到全局 discover 视图。
- 取舍：坚持过滤语义更符合用户心智，也更安全；代价是初次使用时可能看到“空”，但这是正确且可解释的空。

## 组件与接口

### 1. `main.rs` 控制台参数解析

- 职责：识别 `--console` 路径上的目标过滤参数
- 输入：CLI args
- 输出：控制台启动配置，包括过滤关键字列表
- 变更点：新增过滤参数解析与测试覆盖

### 2. `prismtrace_host::console::ConsoleFilter`

- 职责：表达过滤关键字列表与匹配语义
- 输入：target 的 display name、进程名或等价标识
- 输出：是否命中过滤条件
- 变更点：新增匹配辅助函数，供 target/activity/request/health 聚合共享

### 3. 控制台聚合与页面渲染

- 职责：在统一过滤上下文下生成页面和 API payload
- 输入：已有控制台 snapshot 与过滤配置
- 输出：裁剪后的 targets / activity / requests / health，以及过滤上下文摘要
- 变更点：相关空态语义和页面提示需要与过滤上下文保持一致

## 数据与行为变化

建议在控制台读模型中补充：

- `active_filters`
  - 当前生效的过滤关键字列表
- `is_filtered_view`
  - 当前是否为受限范围视图
- `empty_state_reason`
  - 当无匹配目标时，明确表达“当前过滤条件下没有匹配目标”

行为约束：

- 未传过滤参数时，控制台继续表现为默认全局视图
- 传入过滤参数时，Targets / Activity / Requests / Health 必须共用同一过滤结果
- Request Detail 不得暴露未匹配目标的 request

## 正确性与需求映射

### 属性 1：过滤入口必须可显式表达且可测试

- 说明：如果启动参数语义不稳定，用户无法可靠地进入受限范围视图。
- 覆盖需求：控制台启动时必须支持限定监控目标
- 验证方式：参数解析单测覆盖单关键字、多关键字和未传参数

### 属性 2：控制台所有首页聚合区域必须共享同一过滤范围

- 说明：如果 Targets 过滤了但 Activity / Requests 没过滤，页面会失去一致性。
- 覆盖需求：控制台必须只展示匹配过滤条件的目标与聚合结果
- 验证方式：聚合层测试和页面渲染测试覆盖匹配与非匹配目标混合场景

### 属性 3：无匹配时必须给出解释性空态

- 说明：无匹配并不是错误，但必须被用户看懂。
- 覆盖需求：无匹配目标时控制台必须保留可解释空态
- 验证方式：无匹配场景下主页与 `/api/*` payload 保持过滤上下文和空态说明

## 错误处理与降级策略

- 过滤参数解析失败时，host 返回明确错误，不静默忽略非法输入。
- 过滤条件未命中任何目标时，控制台仍正常启动，但各区域使用一致的过滤空态。
- 控制台某个局部数据源暂时不可用时，仍按当前过滤上下文降级，而不是切回全局结果。

## 验证策略

- 单元验证：参数解析、过滤匹配和过滤后聚合逻辑
- 集成验证：控制台首页与 `/api/*` 在过滤上下文下返回一致结果
- 黑盒验证：带过滤关键字启动、无匹配空态、未传过滤参数默认行为
- 回归重点：默认 `add-local-console` 行为不回退，过滤不会误改变 attach / detach 范围

## 风险 / 取舍

- [风险] 过滤字段选择不稳定，导致目标匹配不符合用户预期
  - 通过优先使用 display name 和等价可见标识，并补充测试来降低风险
- [风险] 局部区域忘记接入过滤条件，导致页面不一致
  - 通过共享过滤配置与统一聚合路径降低风险
- [风险] 无匹配时被误实现为“自动回退全局”
  - 在 spec、tasks 和黑盒测试中明确禁止该行为

## 发布 / 回滚

- 发布方式：作为 `prismtrace-host -- --console` 的增量能力上线
- 不涉及存储迁移
- 若实现失败，可回滚过滤参数解析与过滤聚合逻辑，不影响默认控制台路径

## 开放问题

- 过滤参数最终使用的 flag 形式是重复参数还是逗号分隔参数更合适
- 目标匹配是否需要同时考虑 `command_line` 等更细粒度上下文
- 后续如果要支持 UI 内切换过滤条件，是否复用同一过滤配置模型
