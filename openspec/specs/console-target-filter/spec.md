## 变更概述

`console-target-filter` 能力为 PrismTrace 本地控制台增加启动时目标过滤能力，使用户可以在打开控制台时就将观测范围限定到特定 AI 应用，而不是被默认的全局 discover 结果淹没。这个能力关注的是“只看我关心的目标”，而不是在本轮引入完整的动态筛选 UI 或更复杂的会话管理能力。

本次变更优先约束控制台启动参数、匹配语义、过滤后的首页可见行为，以及无匹配时的结果表达。它不要求改变底层 observer source 能力本身，也不要求把过滤器做成控制台内可交互切换的产品功能。

## 术语说明

- 目标过滤条件：用户在启动控制台时传入的一个或多个关键字，用来限制控制台聚合和展示的目标范围。
- 匹配目标：显示名、进程名或等价目标标识中命中过滤条件的本地 AI 应用目标。

## ADDED Requirements

### Requirement: 控制台启动时必须支持限定监控目标
PrismTrace host MUST 在本地控制台启动路径上支持目标过滤参数，使用户可以在打开控制台时，将监控范围限定为一个或多个指定目标关键字。

#### Scenario: 用户以单个目标关键字启动控制台
- **WHEN** 用户使用带有单个目标关键字的控制台启动命令
- **THEN** 控制台以该关键字作为当前监控范围启动，而不是退回默认全局扫描视图

#### Scenario: 用户以多个目标关键字启动控制台
- **WHEN** 用户在控制台启动命令中提供多个目标关键字
- **THEN** 控制台将这些关键字共同视为当前监控范围，并允许匹配任一关键字的目标进入控制台视图

### Requirement: 控制台必须只展示匹配过滤条件的目标与聚合结果
本地控制台 MUST 将目标过滤条件应用到首页可见的观测结果中，使 targets、activity、requests 和 observability health 仅反映匹配目标的范围，而不是继续展示全局 discover 结果。

#### Scenario: target 列表只显示匹配目标
- **WHEN** 用户打开带有目标过滤条件的控制台首页
- **THEN** `Targets` 区域只展示匹配过滤条件的目标，而不展示未匹配的本地目标

#### Scenario: activity 与 request 只聚合匹配目标
- **WHEN** 当前存在同时属于匹配目标与非匹配目标的活动或 request
- **THEN** 控制台只展示与匹配目标相关的活动和 request，而忽略未匹配目标产生的记录

### Requirement: 控制台必须展示当前过滤条件
本地控制台 MUST 在可见界面中展示当前生效的目标过滤条件，使用户知道当前看到的是一个受限范围视图，而不是全局监控结果。

#### Scenario: 首页明确显示当前过滤条件
- **WHEN** 用户打开带有目标过滤条件的控制台
- **THEN** 页面中展示当前过滤条件摘要，且用户无需查看命令行历史也能判断当前监控范围

#### Scenario: 未传过滤条件时不伪装为过滤视图
- **WHEN** 用户以默认方式启动控制台且没有提供目标过滤条件
- **THEN** 控制台不展示误导性的过滤上下文说明，而继续表现为默认全局视图

### Requirement: 无匹配目标时控制台必须保留可解释空态
本地控制台 MUST 在过滤条件没有命中任何目标时继续正常启动，并向用户说明“当前过滤条件下没有匹配目标”，而不是直接回退为全局结果或仅显示空白区域。

#### Scenario: 过滤条件未命中任何目标
- **WHEN** 用户提供的目标过滤条件未匹配任何当前可见目标
- **THEN** 控制台可以正常打开，并以明确空态说明当前过滤条件下没有匹配目标

#### Scenario: 无匹配时不会泄漏全局结果
- **WHEN** 当前系统中存在其他未匹配的本地目标
- **THEN** 控制台仍然不展示这些未匹配目标，而保持与过滤条件一致的空态结果

## 非目标与边界

- 本次不要求改变底层 observer source 的作用范围；过滤能力只约束控制台监控与展示范围。
- 本次不要求在控制台 UI 中提供运行时切换、编辑或保存过滤条件的交互。
- 本次不要求引入正则表达式、复杂查询语法或高级布尔表达式过滤。
